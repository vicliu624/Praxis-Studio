# v0.1 Acceptance Specification

## 1. Open Existing Project acceptance

Using Praxis Studio repository itself:

```text
1. Repository scan succeeds.
2. CodeFactGraphSnapshot is generated and written to rebuildable cache.
3. RepositoryUnderstandingPatch is generated before durable FACT memory is written.
4. FACT memory is written only through explicit acceptance.
5. INFERENCE memory is written.
6. Architecture model candidate is generated.
7. Architecture views are projected from memory/model.
8. Project plan view is projected from specs/model gaps.
9. User can confirm memory records.
10. Confirmed records affect subsequent projections.
11. Explain mode can explain selected architecture node.
12. Plan mode can generate candidate plan.
13. Apply mode can write approved memory/spec/task changes only.
14. Coding task file can be generated.
15. Agent construction emits runtime events.
16. Memory/model/task/trace mutations invalidate affected graph projections.
17. Architecture, project plan, trace and memory views expose fresh / stale / regenerating / failed state.
18. Selecting a graph node / edge / task can produce ContextPacket.
19. Agent uses ContextPacket before repository-wide search.
20. Scope expansion is recorded when anchored context is insufficient.
21. AI code reading produces MemoryPatch / ModelPatch / PlanPatch, not view cache.
22. Dependency view is generated from FACT import/package memory.
23. Architecture component view is generated from ArchitectureModel.
24. UML-like symbol view is generated from UmlModel when symbol extraction is available.
25. Anti-pattern findings are written as structured memory.
26. Basic findings are projected into graph views as annotations.
27. Selecting a finding can open finding-anchored chat.
28. A finding can resolve to an applicable governance playbook.
29. The remediation panel recommends one default path and exposes user intervention points.
30. Praxis can expose status, ContextPacket and task generation through MCP without requiring Desktop UI.
```

## 2. Create New Project acceptance

```text
1. User enters product idea.
2. Idea memory is written.
3. Product model candidate is generated.
4. Domain model candidate is generated.
5. User confirms core distinctions.
6. Architecture model candidate is generated.
7. Specs are generated from confirmed memory/model.
8. Graph views are projected.
9. Project plan is generated.
10. Skeleton is created only after confirmation gates.
11. TASK-0001.md is generated.
12. Projection views update when confirmed memory or task state changes.
13. Generated task can be selected as a context anchor.
14. Task anchor exposes related specs, memory, architecture nodes and source paths when available.
15. Gantt view is generated from PlanModel.
16. Task progress suggestions are CANDIDATE until confirmed.
17. Quality findings can generate plan actions or tasks.
18. Governance playbooks can recommend conservative task/spec/model improvements before source changes.
```

## 3. Quality management acceptance

v0.1 should detect at least these basic anti-patterns:

```text
1. Graph node without sourceMemoryIds.
2. Task without acceptance criteria.
3. Task without dependency or deliverable.
4. Projection stale.
5. Package dependency cycle.
6. Architecture dependency without evidence.
7. Candidate used as confirmed.
8. Agent repeated read or context drift.
```

For each detected finding, Praxis must:

```text
write finding memory
show graph annotation or Quality Inbox item
allow finding-anchored chat
select an applicable governance playbook when available
recommend one remediation path rather than equal-looking options
expose user intervention points for meaning, risk, priority, scope and strength
record status changes
rerun detector after resolution attempt
refresh affected graph projections
```

## 4. Fixture and contract acceptance

At least one automated fixture suite must prove:

```text
schema round-trip validation
intake output stability
projection manifest status handling
finding/playbook selection stability
```

See `docs/specs/27-test-fixtures-and-golden-files.md`.

## 5. Negative acceptance

Praxis v0.1 must not:

```text
- treat graph nodes as source of truth
- treat unconfirmed Agent output as confirmed memory
- generate code directly from raw idea
- modify existing source code automatically
- hide whether a relation is FACT or INFERENCE
- generate graph nodes without source memory IDs
- silently update graph views without memory or trace source
- present stale projections as fresh
- let projection cache become source of truth
- search the wider repository before using available ContextPacket
- hide scope expansion from trace or transcript
- let a task anchor authorize unrelated code changes
- let provider scratch or cache become durable project authority
- let AI directly edit views/**/*.json or views/**/*.mmd
- treat Mermaid / Gantt JSON as source of truth
- let AI write FACT or CONFIRMED memory
- accept model patches without source evidence
- show graph warning without explainable finding source
- mark AI-detected finding as confirmed without user approval
- resolve finding without evidence or detector rerun
- hide findings only because they are inconvenient
- force users to design architecture remediation from scratch
- present every remediation alternative as equally recommended
- let prompt templates bypass governance playbooks
```
