C4 Component 用来解释“Container 内部哪些关键组件承担架构职责”。

Component 必须隶属于某一个明确的 Container。它是该 Container 内部的主要结构单元、接口单元或协作单元，不是全量类图、函数清单、领域对象清单或内部扫描指标排行榜。

## 必须解释

- 组件角色：入口、应用服务、领域服务、接口、端口、适配器、仓储、网关、客户端、注册表、配置、UI 页面/面板、worker 或清晰协作单元。
- 组件之间的关系意义：编排、调用、依赖反转、事件、注册、渲染、持久化、转换或策略选择。
- 为什么这个对象是架构组件，而不只是普通函数/类/文件。
- 它如何承接业务能力，以及它的变更会影响哪些 Container 或 Use Case。
- 下钻到 Code View 时需要查看哪些代码锚点。

## 不要做

- 不要输出全量类图。
- 不要把普通值对象、DTO、Request/Response、Exception、Enum、Money/Amount/Id、工具方法、孤立方法或普通领域实体放进 Component View，除非它们在当前 Container 中承担明确接口、策略、服务、适配、端口或协作职责。
- 不要把关系数字当作解释本身；需要解释它为什么代表共享核心、编排锚点、适配边界或变更影响面。
- 不要用“同一边界内的关键组件”这类空泛句子替代具体职责。
- 不要使用 `fanIn`、`fanOut`、事实图 id 或内部分析节点名作为用户可见解释；应改写成“被多个入口复用”“向多个外部能力编排”等业务可读描述，并说明依据。
