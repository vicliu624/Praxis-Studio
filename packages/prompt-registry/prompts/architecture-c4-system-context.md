C4 System Context 用来解释“这个系统和外界如何协作”。

它的视角是目标软件系统作为一个黑盒所处的环境。System Context 不是业务流程图，不是模块依赖图，也不是部署图。

## 必须解释

- 主要人类角色是谁，他们为什么和系统交互。
- 外部系统、第三方服务、上游/下游系统、真实人类角色分别承担什么责任；这些边界必须来自目标项目证据或已确认文档。
- 系统与外部对象之间的关系意义：读取、写入、调用、确认、治理、执行或反馈。
- 哪些内容属于系统内部，哪些内容必须留在外部上下文。
- 这个上下文如何帮助读者进入 Container 层。
- 如果当前目标系统的真实外部参与者或外部系统证据不足，必须明确写成“未命名外部参与者 / 未命名外部系统”和证据缺口；不要用工具、IDE、LLM、Git 或文档生成流程填空，除非目标项目本身就是这些系统。

## 不要做

- 不要展开内部模块、类、函数或文件。
- 不要展示目标系统内部的 Container、Component 或 Code 元素；下钻入口可以在文字或索引里出现，但 System Context 图本身必须保持黑盒。
- 不要把系统上下文写成技术依赖图。
- 不要把用户故事细节塞进 System Context。
- 不要把 Praxis Studio、LLM Provider、外部 coding agent、生成文档、Git 仓库或 IDE 工作流当作目标项目的 System Context 元素，除非目标项目本身就是这些工具/服务并且仓库证据能证明。
- 不要用工具侧角色填补证据缺失的业务角色；证据不足时写成“未命名外部参与者”或“未命名外部系统”，并在“判定依据”中说明当前仓库缺少哪些目标项目证据。禁止写“待确认”“后续 agent 确认”或要求用户替 agent 完成事实判定。
