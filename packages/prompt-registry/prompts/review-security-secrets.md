## 反误报规则

- 不要因为源码类名、方法名或领域概念里出现 `Token`、`Credential`、`Authorization`、`Auth`、`Password` 等词，就判定存在敏感文件或密钥泄露。业务模型、DTO、命令、响应对象和领域值对象本身不是密钥材料。
- 只有当证据指向真实密钥/证书文件、环境配置、发布配置、密钥目录、证书目录、`.env`、`.npmrc`、`appsettings.Production.json` 或包含明文敏感值的配置项时，才输出安全与敏感信息候选问题。
- `*.java`、`*.ts`、`*.tsx`、`*.kt`、`*.cs` 等源码文件只有在内容证据明确包含明文密钥、证书正文、密码、私钥、生产 token 或外部系统凭据时，才能作为敏感信息问题证据；不能只凭文件名。
- 如果证据只是“有支付凭证/登录令牌相关业务代码”，应改写为业务建模或接口契约问题，不能放入安全敏感信息分类。
- `META-INF/services/*`、数据库 changelog、migration、schema 迁移脚本中出现 Token/Auth/Credential 名称时，默认表示类型名、表名或变更名；除非内容证据包含明文 secret、password、private key、client secret 或真实证书材料，否则不要判定为敏感信息泄露。
